把 Arthas 交给 AI：集成 Claude Code CLI 与 Arthas MCP 的实战复盘

标签： Engineering

在复杂的 Java 生产环境下，传统的 Arthas 交互模式（SSH 登录 -> 挂载 -> 手敲 trace/watch）虽强大，但在高频问题排查中仍存在效率瓶颈。随着 Model Context Protocol (MCP) 的成熟，Arthas 4.x 引入了对 MCP 的支持，使得 AI 助手（如 Claude Code CLI）能够直接调度其底层的诊断技能（Skills）。

本文涉及的主要依赖版本如下：

  • Arthas：4.1.8
  • Claude Code CLI：v2.1.91

上下文与问题定义 (Context & Problem Definition)

我的初衷是构建一个“AI 驱动的诊断控制台”，让 Claude Code 直接执行 arthas中 thread、jad 等操作。然而，在初步尝试挂载工具时，我遭遇了两个核心障碍：

通过翻阅 Arthas 的 Configure 类及其 HTTP 处理逻辑，我发现 MCP 服务并非通过命令行参数（CLI Options）开启，而是作为 HTTP Server 的一个特殊路径。

  • 源码发现：McpHttpRequestHandler 负责拦截符合 mcpEndpoint 配置的请求。
  • 配置优先级：该路径必须在 arthas.properties 或系统变量中显式定义，否则默认不激活。

验证桥接方案的可行性

由于 Claude Code 需要 Stdio 接口，我提出“远端转本地”的假设：寻找一个通用的 MCP 桥接器（Bridge），将远端 SSE 流转换为 Stdio 流。经过对比，社区的 mcp-remote 工具符合这一架构要求。
