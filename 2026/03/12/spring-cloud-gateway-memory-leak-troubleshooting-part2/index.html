Spring Cloud Gateway 堆外内存泄漏排查（下）

标签： Engineering

本文是 Spring Cloud Gateway 堆外内存泄漏排查（上） 的续篇。在上篇中，我定位到了自定义 Filter 中使用 .buffer() 操作符导致堆外内存泄漏的问题，并将方案调整为 DataBufferUtils.join()。

泄漏再次发生

在首次修复上线一周后，同样的告警再次触发。

Netty 的 ResourceLeakDetector 再次输出了泄漏报告：

查看内存监控数据，发现 Direct Buffer 的增长速度确实比修复前慢了很多，但仍然在缓慢增长。

从错误信息可以得出两个结论：

  1. 首次修复有效果：之前 2 天出现 17 条 leak records，现在 7 天仅出现 7 条 leak records，泄漏频率明显降低。
  2. 问题仍然存在：泄漏位置依然是处理下游微服务响应的阶段（原因分析见上篇）。

本次修复的核心改进：

  1. 使用 try-finally 确保资源释放：无论 map 操作中是否抛出异常，finally 块中的 DataBufferUtils.release(dataBuffer) 都会被执行。

反思与收获

本次排查沉淀的核心教训：

Netty 池化直接内存的生命周期完全依赖引用计数机制，而在 WebFlux 场景下操作 DataBuffer 时，必须严格遵循”谁消费、谁释放”的核心原则。

更重要的是，不能仅关注正常流程的资源回收，还需重点考量以下异常场景的清理逻辑：

  1. 流被提前取消：如客户端中断请求、下游服务超时等场景。
  2. 处理过程抛出异常：必须使用 try-finally 或 Reactor 的错误处理钩子确保资源释放。

响应式流的异常处理机制与 WebFlux 底层运行原理决定了：无论是日常开发还是问题排查，”消费即释放”的原则与流中断时的资源兜底清理，都是必须时刻绷紧的”生命线”，稍有疏忽便可能引发内存泄漏等隐性问题。
