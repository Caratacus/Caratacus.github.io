基于 Spotless + Git Hook 的 Java 代码格式化尝试

Category: Engineering

在日常的 Java 开发中，团队协作时往往需要保持代码风格的统一。随着 AI 辅助编程工具（如 Claude Code、Cursor）的使用，我发现有时生成的代码在缩进或导入（import）上可能不完全符合现有的项目规范。

为了减少每次提交代码前手动运行格式化命令的繁琐步骤，我尝试通过 Gradle 加上 Spotless 插件以及 Git Pre-commit Hook，配置了一套自动触发的代码格式化流程。在此记录一下我的配置过程。

格式化触发时机的选择

最初，我尝试将格式化任务（如 spring-javaformat 或 spotlessApply）直接绑定到 Gradle 的 compileJava 任务上：

但我发现，在较新的 Gradle 版本（8.x/9.x）中，这种做法可能会引发隐式依赖（Implicit Dependency）报错。而在涉及到 Spring Boot AOT 预编译时，甚至可能会出现循环依赖（Circular Dependency）。

因此，我改变了策略，将格式化动作从构建（Build）阶段剥离，尝试转移到版本控制（Git Commit）阶段。

总结反思

通过 Spotless + Pre-commit Hook 的组合，代码格式化变成了提交过程中的一个自动环节。我发现这种方式有几个好处：

  1. 减少手动操作：无论是在 IDE 中提交还是使用命令行，Hook 都会在后台执行格式化。
  2. 与 AI 工具配合：当 AI 助手生成代码后尝试提交时，Hook 也能自动修正潜在的缩进和 import 问题。
  3. 环境适应性：脚本中加入了对任务是否存在的判断，如果环境未配置 Spotless，它会静默跳过而不会报错阻塞提交流程。

这次配置实践帮助我更好地理解了 Gradle 任务的依赖关系以及 Git Hook 的工作机制，也为后续进一步探索工程化自动化提供了参考。
